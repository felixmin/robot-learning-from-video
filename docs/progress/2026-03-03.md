# 2026-03-03

## Dataset unification exploration

Objective for today:

- decide whether HLRP should stay on raw OXE, build a mixed raw-OXE-plus-LeRobot stack, or move toward a LeRobot-v3-first dataset stack,
- extract the actual dataset contract required by the codebase,
- document a migration path that keeps Stage 1, Stage 2, and Stage 3 aligned.

## Repo findings

### Current active path

- `packages/common/data_factory.py` currently only supports `data.backend=oxe_local_indexed`.
- Stage 1 still consumes `frames + metadata`.
- Stage 2 already normalizes into `FoundationBatch`, but only after going through an OXE-specific batch path.
- Stage 3 is already LeRobot-native and converts LeRobot batches into `FoundationBatch`-compatible inputs.

### Practical implication

The real abstraction gap is below Stage 1 and below `FoundationBatch`.

The clean solution is:

1. one HLRP temporal-window dataset contract,
2. one HLRP weighted mixer,
3. format-specific source adapters under that contract.

## Decision after today’s exploration

Preferred path:

1. move toward official LeRobot v3 datasets as the long-term dataset format,
2. implement an HLRP-owned temporal-window interface and weighted mixer,
3. keep the raw OXE loader only as a temporary comparison/fallback path,
4. do not invest in a permanent mixed raw-format pipeline.

## Plan refinement after design review

After re-checking the repo and LeRobot integration points, the migration plan was tightened in a few places to keep v1 simpler.

Changes to the plan:

- make image resizing explicit in the request/source contract,
- keep source output as resized `uint8` tensors,
- keep normalization in Stage 1 / Stage 2 adapters or downstream backends,
- remove `mix_block_size` from the initial design,
- remove worker-runtime eviction policy from the initial design,
- define validation as finite and deterministic per-source held-out episodes,
- keep source weights in mixer config, not in compiled metadata indices,
- treat action/state normalization as an adapter-stage concern, not a dataset-source concern,
- add quantitative parity checks instead of only qualitative comparison language.

Normalization decision after follow-up review:

- keep one global normalization object per training run,
- do not use per-dataset normalization,
- use raw canonical action/state tensors through the dataset interface,
- for single-source runs, use that source's `meta.stats`,
- for mixed runs, build one weighted merged stats object from the selected sources' `meta.stats`,
- weighting of merged stats should match the configured mixture, not only raw dataset sizes.

Test and benchmark planning decision:

- keep correctness and interface-contract checks in `tests/`,
- keep throughput measurement in `experiments/benchmarking/`,
- use synthetic pytest fixtures for most coverage,
- keep live LeRobot dataset smoke tests opt-in via environment flag,
- add one dedicated LeRobot-v3 dataloader benchmark script with batch-size / worker-count sweeps and JSON/CSV output.

Rationale:

- current OXE code already resizes in the dataset/adapter path,
- Stage 2 already does model-specific normalization downstream,
- Stage 2 and Stage 3 both already fit a single shared stats object better than source-dependent normalization,
- the repo already separates correctness tests and benchmarking scripts this way,
- locality knobs and eviction logic are optimization concerns and should only be added after profiling,
- explicit validation semantics and action normalization boundaries are required to make the migration implementable without hidden assumptions.

## Mixer survey

### LeRobot built-in multi-dataset path

Current vendored LeRobot code does contain `MultiLeRobotDataset`, but it is not a real weighted mixer and it is not wired into training.

Observed behavior:

- `lerobot/src/lerobot/datasets/lerobot_dataset.py`
  - `MultiLeRobotDataset` constructs one `LeRobotDataset` per repo id.
  - it concatenates them into one index space.
  - it keeps only the intersection of feature keys across all datasets.
  - it injects `dataset_index` into each item.
- `lerobot/src/lerobot/configs/default.py`
  - dataset config comments explicitly describe list-valued `repo_id` as concatenation.
- `lerobot/src/lerobot/configs/train.py`
  - `TrainPipelineConfig.validate()` raises `NotImplementedError` when `dataset.repo_id` is a list.
- `lerobot/src/lerobot/datasets/factory.py`
  - `make_dataset()` also raises `NotImplementedError` before the `MultiLeRobotDataset` branch can run.

Implication:

- built-in LeRobot multi-dataset support is currently concatenation-oriented and effectively disabled for training,
- it does not provide explicit source weighting,
- it is not sufficient for HLRP’s desired mixture control.

### External mixer patterns

#### OpenVLA

Source files:

- `https://github.com/openvla/openvla/blob/main/prismatic/vla/datasets/rlds/oxe/mixtures.py`
- `https://github.com/openvla/openvla/blob/main/prismatic/vla/datasets/rlds/dataset.py`

Observed pattern:

- define named mixtures as `(dataset_name, weight)` lists,
- compute dataset sizes once,
- optionally multiply weights by dataset sizes with `balance_weights`,
- repeat each source dataset indefinitely,
- apply trajectory transforms per source,
- flatten trajectories to frames,
- interleave the repeated frame streams with `DLataset.sample_from_datasets(datasets, sample_weights)`,
- shuffle after interleaving,
- allocate read/transform threads across datasets according to mixture weights.

Implication:

- OpenVLA mixes at the frame-stream level, not by concatenation,
- it treats weights as a first-class research parameter,
- it optimizes input pipeline resources around the configured mixture.

#### Octo

Source files:

- `https://github.com/octo-models/octo/blob/main/octo/data/oxe/oxe_dataset_mixes.py`
- `https://github.com/octo-models/octo/blob/main/octo/data/dataset.py`

Observed pattern:

- same overall structure as OpenVLA,
- named OXE mixtures are declared explicitly with per-dataset weights,
- `make_interleaved_dataset()` computes sizes, optionally balances weights by size, allocates threads by weights, repeats each source, flattens to frames, then uses `sample_from_datasets(...)`,
- applies frame transforms after interleaving,
- sets a small RAM budget on the iterator pipeline.

Implication:

- Octo and OpenVLA converge on the same design:
  - explicit named mixtures,
  - optional size balancing,
  - interleaving repeated per-dataset streams rather than simple concatenation.

#### Generic iterable mixing

Reference:

- `https://github.com/webdataset/webdataset/blob/main/src/webdataset/mix.py`

Observed pattern:

- `RoundRobin` and `RandomMix` provide iterable-level round-robin or probability-based source mixing,
- there is no robotics-specific handling for episodes, action windows, camera keys, or temporal validity.

Implication:

- generic data-loader libraries solve the source-selection part,
- robotics projects still need a domain-specific layer above that for temporal sampling and modality alignment.

#### Third-party LeRobot-specific weighted sampler

Reference:

- `https://pypi.org/project/robocandywrapper/`

Observed package behavior from `robocandywrapper==0.2.7`:

- package summary: "Sweet wrappers for extending and remixing LeRobot Datasets"
- ships `robocandywrapper/samplers/weighted.py`
- ships `robocandywrapper/samplers/factory.py`
- sampler config supports:
  - `dataset_weights`
  - `samples_per_epoch`
  - `shuffle`
  - `replacement`
  - optional episode selection metadata
- weighting semantics:
  - each dataset gets an effective size of `dataset_length * weight_multiplier`
  - per-dataset sample counts are computed once per epoch from the normalized effective sizes
  - flat indices are then drawn within each dataset range, concatenated, and globally shuffled

Important limitations:

- this is not upstream LeRobot,
- the package contents visible in the wheel are centered on `lerobot_21`, not LeRobot v3,
- the sampler expects a wrapper exposing flat dataset ranges and mixes at the concatenated-index level,
- it does not appear to handle request-dependent temporal validity, per-episode locality, or multi-camera temporal-window semantics.

Implication:

- yes, at least one third-party LeRobot-oriented weighted sampler exists,
- but it does not look like a mature LeRobot-v3-native solution that HLRP can adopt directly.

### HLRP-specific conclusion from the survey

There are two relevant precedents:

1. LeRobot’s built-in path is too weak for HLRP because it is concatenation-only and training-disabled.
2. OpenVLA and Octo both show that explicit weighted interleaving is the standard solution when mixing many robotics datasets.

For HLRP, the right design is still slightly different from both:

- keep the existing HLRP sampler idea of `dataset -> episode -> anchor`,
- keep explicit per-dataset weights,
- keep deterministic reshuffling/cycle behavior,
- do not rely on feature intersection as the normalization mechanism,
- return one HLRP-owned temporal-window sample type instead of raw backend items.

Why not copy OpenVLA/Octo literally:

- they operate in RLDS/TensorFlow pipelines and interleave repeated frame streams directly,
- HLRP needs request-dependent temporal validity for Stage 1 frame pairs and Stage 2 action chunks,
- HLRP already has a stronger sampler precedent in `openx_local_indexed_full.py` for weighted dataset/episode sampling.

So the external survey strengthens, not weakens, the current recommendation:

- keep the HLRP mixer owned by HLRP,
- make it LeRobot-v3-first,
- borrow explicit mixture registries and optional size-balancing concepts from OpenVLA/Octo,
- do not use `MultiLeRobotDataset` as the production solution.

## Plan refinement

The plan document was expanded from a migration note into an implementation-grade design.

Main additions:

- explicit proposed module/function/class names for the new LeRobot-v3-first backend,
- canonical `DatasetRequest` / `DatasetSample` / `BatchedDatasetSample` / `SampleToken` interfaces,
- metadata-only compiled indexing plan,

- distributed-sampling semantics,
- cache ownership and persistence notes,
- RAM/CPU overhead estimates and expected bottlenecks.

Important new constraint captured in the plan:

- do not instantiate many filtered `LeRobotDataset(..., episodes=[...])` subsets for runtime mixing,
- keep one `LeRobotDataset` per source per worker,
- sample by absolute anchor index against a metadata-compiled episode index instead.
- after clarifying Stage 1 requirements, the plan now treats a dedicated `Stage1Batch` as the target consumer interface rather than preserving the legacy single-`frames` dict as the long-term shape.

## Temporal-query capability note

LeRobot itself already supports the retrieval side of more general Stage 1 requests:

- multiple camera keys,
- arbitrary temporal offsets via `delta_timestamps`,
- action windows,
- task text,
- per-key pad masks.

What is still limited today is not LeRobot's data format, but the current online-LAQ consumer code.

Current Stage 3 status:

- generic LeRobot data loading can return arbitrary temporal windows for observation keys,
- current `latent_smol` online-LAQ path in `lerobot_train.py` uses a single `laq_camera_key` and assumes exactly two frames,
- current HLRP Stage-3 policy code is partially generalized to multiple `laq_camera_keys`, but still rejects anything other than exactly one camera and exactly two observation steps when actually computing online latent targets.

Implication:

- the planned HLRP dataset contract should remain general over camera count and temporal length,
- but Stage 1 and Stage 3 LAQ-consumer code will need a separate model/adapter update to consume multi-camera or longer temporal sequences.
